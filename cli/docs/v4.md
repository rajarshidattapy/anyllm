# PRD: Cross-CLI Integration System for anyllm

## Status

Proposed

## Author

Rajarshi Datta

## Goal

Enable `anyllm` to integrate natively with every major AI coding CLI so users can seamlessly migrate context between agents when they hit rate limits, provider outages, or simply prefer another model.

The `.anyllm` directory becomes the portable source of truth, while each CLI receives native commands (slash commands, skills, prompts, etc.) that expose the same `anyllm` workflows.

---

# Vision

> Install once. Integrate everywhere. Carry context across agents.

Example workflow:


The project memory should follow the user regardless of which coding agent is currently active.

# Problem Statement

Today:

```text

‚îú‚îÄ‚îÄ creates .anyllm
‚îî‚îÄ‚îÄ installs Claude-specific commands
```

Problems:

* tightly coupled to Claude Code
* difficult to support new coding agents
* portability suffers
* initialization and integrations are mixed responsibilities
* impossible to provide a universal migration experience

---

# Product Principles

### 1. Project memory is universal.

`.anyllm` must work everywhere.

### 2. Integrations are optional.

CLI-specific commands are convenience wrappers.

### 3. Use native extension mechanisms.

Do not force every agent into the same plugin architecture.

### 4. Preserve muscle memory wherever possible.

Users should use the same concepts:

```text
pack
prime
push
repack
status
```

even if invocation syntax differs.

# Installation Modes

## Global

```bash
anyllm integrate --all --global
```

Installs:

```text
~/.claude/commands/
~/.gemini/config/skills/
~/.config/opencode/commands/
~/.codex/prompts/
```

---

## Project

```bash
anyllm integrate --all --project
```

Installs:

```text
.agents/skills/
.opencode/commands/
```

inside repository.

This allows:

```bash
git clone repo
cd repo
codex
```

Immediate access to:

```text
$anyllm-pack
$anyllm-prime
```

without additional setup.
